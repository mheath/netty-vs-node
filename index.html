Classic Thread-per-connection Architecture

Thread-per-connection

  • Response to multi-process topology
  • Threads have lower overhead than processes
  • Threads maintain their stack/environment
  • Threads switching is relatively expensive
  • Apache, Tomcat, JBoss, WebSphere, WebLogic, etc.

Blocking

  • Blocked threads maintain state/environment
  • Blocked threads use memory
  • Hardware vendors love this model
  • They sold it to you (IBM, BEA, Sun/Oracle)

Event-driven Architecture

  • Response to thread-per-connection architecture
  • Single thread handles multiple network connections
  • Uses reactor pattern demultiplexing (for your boss)
  • Nginx, Twisted, EventMachine, Netty, Node.js, etc.

Node.js

  • Node is the V8 JavaScript engine outside of the browser
  • Has additional APIs for doing network and file I/O
  • Most commonly used for building event-driven network applications

Netty

  • JBoss project for building event-driven network applications
  • Runs on the JVM
